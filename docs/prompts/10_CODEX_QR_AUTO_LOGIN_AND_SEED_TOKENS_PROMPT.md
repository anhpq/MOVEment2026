# 2. Confirmed Feature Goal

Each active Team must have one active Automatic URL QR Login token.

Official QR payload:

```text
{FRONTEND_PUBLIC_URL}/qr-login?token={RAW_TEAM_QR_TOKEN}
```

Expected flow:

```text
Team scans QR
→ browser opens frontend route
→ frontend reads token
→ frontend removes token from visible URL
→ frontend calls backend QR Login endpoint
→ backend validates token and Team
→ backend creates normal Team session
→ backend revokes previous active Team session
→ frontend stores auth state
→ frontend loads current Team
→ frontend redirects to normal Team destination
```

The Team must not be required to type:

- username;
- password;
- Team code;
- raw QR token.

# 3. Confirmed Business Rules

Implement according to these rules.

## 3.1 QR Format

Official format:

```text
{FRONTEND_PUBLIC_URL}/qr-login?token={RAW_TEAM_QR_TOKEN}
```

The QR must contain a frontend URL, not a backend API URL.

## 3.2 Token Security

The raw token must be:

- random;
- opaque;
- unique;
- generated by a Cryptographically Secure Random Generator;
- at least 128 bits of entropy;
- not derived from `teamId`;
- not derived from username;
- not derived from Team name;
- free of plaintext credentials.

Do not generate new tokens such as:

```text
MV26-TEAM-01-LOGIN
MV26-TEAM-02-LOGIN
```

These are Legacy formats only.

## 3.3 Reusable Controlled Token

The active Team QR Login token is reusable while valid.

Successful login must not permanently consume the token.

The same active token may be used again when the Team needs to log in on another device or re-enter the application.

The token must support:

- active/inactive state;
- revoke;
- rotate;
- expiry or Event validity;
- usage auditing;
- rate limiting.

## 3.4 One Active Session per Team

Password login and QR Login use the same Team session policy.

A successful new Team login must revoke the previous active Team session.

QR token lifecycle and Team session lifecycle are separate.

Do not revoke the QR token merely because a Team session expires or is replaced.

## 3.5 Automatic Provisioning

When a new Team is created, the backend must automatically provision its required Team QR Login token.

The user must not be required to invent a raw token.

## 3.6 Seed Repair

Local/test seed or a maintenance command must create an active QR token for a Team that is missing one.

Repeated seed runs must preserve an existing valid active token.

## 3.7 Production Safety

Do not automatically print raw Production QR tokens into:

- CI/CD logs;
- deployment logs;
- committed Markdown;
- committed JSON;
- tracked seed output;
- application logs.

## 7.4 Atomic Operations

# 8. Token Generation

Use a CSPRNG.

Node.js example approach:

```text
crypto.randomBytes(...)
```

Recommended:

```text
Base64URL
minimum 128 bits entropy
```

Do not include:

- Team ID;
- Team number;
- username;
- password;
- timestamp as the only entropy;
- predictable prefix that reveals identity.

A non-secret display label may be stored outside the token.

---

# 9. Token Lookup and Verification

Recommended flow:

1. validate raw token input;
2. calculate SHA-256 fingerprint;
3. lookup token record by fingerprint;
4. verify secure hash if the project uses secondary verification;
5. check active state;
6. check `revoked_at`;
7. check `expires_at`;
8. check Event validity;
9. check associated Team active state;
10. enforce rate limit;
11. create Team session;
12. update usage metadata.

Do not compare raw tokens from plaintext database columns.

Do not log raw tokens.

# 14. Raw Token and QR Artifact Strategy

Audit which strategy is currently used.

## Strategy A — Encrypted Raw Token

Allowed when:

- encryption key is external;
- Admin access is authorized and audited;
- plaintext is never logged.

## Strategy B — Protected QR Artifact

Allowed when the QR image is stored in protected storage.

## Strategy C — Display Once and Rotate to Reprint

Allowed when only hashes are stored.

In this strategy:

- raw token is displayed only at creation;
- reprint requires token rotation;
- old QR is revoked.

Do not claim a raw token can be recovered from a hash.

Report the selected strategy.

This route must be public:

```text

```

It must not redirect to `/login` before token validation.

## 16.2 Flow

1. read token from query string;
2. retain token in memory;
3. remove token from the visible URL safely;
4. show loading state;
5. call backend once;
6. prevent duplicate effect/request;
7. store auth state using existing auth flow;
8. load current Team;
9. redirect to the same destination as normal Team login.

## 16.3 URL Cleanup

After safely reading the token, the visible URL should become:

```text
/qr-login
```

Use router replace or `history.replaceState`.

Do not leak token to analytics, error breadcrumbs, or external referrers.

## 16.4 Duplicate Protection

Protect against:

- React Strict Mode;
- rerender;
- double effect;
- double-click;
- back/forward navigation;
- simultaneous retry.

Use stable guards such as:

```text
hasStartedRef
requestId
isSubmitting
```

## 16.5 Success

On success:

- store the normal Team auth response;
- load current Team state;
- redirect to normal Team destination.

Do not create a separate QR-only application state.

## 16.6 Failure

Do not silently redirect to `/login`.

Show a safe error and actions such as:

- retry;
- go to login;
- use username/password when supported.
